Factory OS — это не фреймворк и не библиотека. Это набор промптов, правил и конвенций поверх Claude Code, который превращает одного разработчика в команду из 15 специалистов.

## Проблема

AI-ассистенты типа ChatGPT и Copilot помогают писать код. Но они не умеют:

- Декомпозировать задачу на подзадачи
- Делегировать работу разным специалистам
- Контролировать качество результата
- Помнить контекст проекта между сессиями
- Работать автономно без постоянного надзора

Factory OS решает все пять проблем.

## Архитектура

Система состоит из трёх уровней:

### 1. CEO-агент (оркестратор)

CEO получает задачу от меня и декомпозирует её. Он никогда не пишет код сам — только делегирует. У него есть доступ к:

- **DNA проекта** — описание архитектуры, зависимостей, горячих точек
- **Task board** — текущие задачи и их статус
- **Memory files** — решения, принятые в предыдущих сессиях
- **Rules** — жёсткие правила, которые нельзя нарушать

CEO порождает агентов через Agent tool с явным указанием модели, роли и задачи.

### 2. Специализированные роли (15 штук)

Каждая роль — это промпт-файл (`~/.factory/roles/*.md`) с:

- Описанием зоны ответственности
- Разрешёнными и запрещёнными действиями
- Чек-листом качества
- Моделью (Opus для Builder, Sonnet для Strategy)

Основные роли:

| Роль | Модель | Что делает |
|------|--------|-----------|
| Builder | Opus | Пишет код, запускает тесты |
| Discovery | Sonnet | Исследует рынок, анализирует конкурентов |
| Strategy | Sonnet | Продуктовая стратегия, позиционирование |
| Quality | Sonnet | Code review, тестирование |
| DevOps | Opus | Деплой, инфраструктура, мониторинг |
| Marketing | Sonnet | SEO, контент, кампании |
| Designer | Sonnet | UI/UX решения |

### 3. Правила и гейты

Система работает не потому что AI умный, а потому что правила жёсткие:

- **Preflight checklist** — проверки перед каждым действием
- **Quality gates** — тесты должны проходить перед коммитом
- **Session start** — обязательный reload контекста при старте
- **Methodology gates** — DNA, снапшоты, без деградации

Правила хранятся в `~/.factory/rules/` и читаются при каждом запуске агента.

## Как это работает на практике

Пример: "Добавить страницу контактов на nevr.aicpo.ru".

1. Я пишу задачу CEO-агенту
2. CEO читает DNA проекта, понимает что это Astro-сайт
3. CEO порождает Builder-агента с конкретным task table:
   - Создать `src/pages/contact.astro`
   - Добавить навигацию в Header
   - Добавить текст из CEO-указаний
4. Builder читает существующие файлы, пишет код, запускает `astro build`
5. Если build fails — Builder чинит, максимум 3 попытки
6. Builder коммитит с описательным сообщением
7. CEO получает отчёт и переходит к следующей задаче

Весь процесс занимает 2-5 минут для типичной задачи.

## Memory — как агенты помнят

Самая критичная проблема AI-ассистентов — потеря контекста. Claude Code имеет окно в 200K токенов, но один проект может иметь миллионы строк кода.

Factory OS решает это через memory files:

- `MEMORY.md` — индекс всех файлов памяти
- `knowledge/*.md` — фреймворки, методологии, экспертиза
- `feedback_*.md` — уроки из ошибок, правила поведения
- `project_*.md` — состояние каждого проекта

При старте сессии CEO-агент читает все memory files (шаг S1 в Session Start checklist). Это гарантирует, что решения из предыдущих сессий не теряются.

## Feedback loop

Когда агент делает ошибку, я создаю `feedback_*.md` файл с описанием:

```
Правило: Никогда не удалять незакрытые задачи из DISCUSSION.md
Причина: Пользователь потерял 3 задачи, которые агент посчитал решёнными
```

Этот файл читается при каждом следующем запуске. Система учится на ошибках — не через fine-tuning, а через явные правила.

## Результаты

За три недели с Factory OS:

- 9 продуктов от нуля до продакшена
- ~50 000 строк кода написано агентами
- ~500 коммитов
- 0 критических багов в продакшене
- Среднее время от задачи до деплоя: 15 минут

Система не идеальна. Агенты иногда галлюцинируют, иногда ломают то, что работало. Но правила и гейты ловят 95% ошибок до коммита.

## Ограничения

Factory OS не подходит для:

- Команд больше 1 человека (нет мультиплеера)
- Проектов с закрытым кодом (Claude Code видит всё)
- Задач, требующих физического тестирования
- Продуктов без чётких требований

Это инструмент для solo maker, который точно знает, что хочет построить, и хочет построить это быстро.
